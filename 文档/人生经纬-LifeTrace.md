# 人生经纬 LifeTrace 设计规格

日期：2026-07-16

状态：规格已确认，进入实施
目标平台：Codex/OpenClaw Skill、ClawHub、GitHub

## 1. 产品定义

“人生经纬 · LifeTrace”把人物经历转换为可核验、可播放、可探索的动态人生足迹地图。它面向历史人物、公众人物和用户主动提供资料的普通人，输出一个可直接在浏览器中查看的 HTML 作品。

核心价值不是简单地把地点连线，而是把“何时、何地、发生了什么、依据是什么、地点有多确定”统一在同一条时空叙事中。

首个 demo 使用刘备。视觉上融合两张参考附件：采用附件一的全国尺度密集迁徙红线和节点表达，并使用附件二的深色叙事舞台、地点故事卡与时间轴结构。

## 2. 使用场景

1. 历史人物：输入“生成刘备一生足迹图”，自动整理公开史料、地点和年代，标记来源及可信度。
2. 公众人物：从公开可靠资料中提取求学、工作、创作、迁居和重要活动地点。
3. 普通人：上传 JSON 或 CSV，不主动搜索私人信息，生成个人、家族或纪念性人生地图。
4. 教学与展览：按时期播放迁徙，点击地点讲述事件，用于历史课堂、人物馆和数字人文展示。
5. 内容创作：生成可托管到 GitHub Pages 或嵌入展示页面的互动 HTML。

## 3. 产品边界

### 包含

- 名人/历史人物的公开资料搜索、结构化和来源记录。
- 普通人 JSON/CSV 导入。
- 高德地点搜索、地理编码和地图展示。
- 时间排序、地点去重、置信度分级、动态路线播放。
- 地图与右侧故事时间轴双向联动。
- 可重复生成的独立 HTML demo。
- GitHub 与 ClawHub 所需的 Skill 内容和发布说明。

### 不包含

- 根据零散信息推断普通人的住址或实时位置。
- 把现代驾车路线表述成古代真实行军路线。
- 无来源的精确日期、坐标或事件细节。
- 用户账号、云数据库、在线编辑器和多人协作。
- 自动替用户发布到外部平台；交付物只准备到可发布状态。

## 4. 总体方案

采用混合生成式架构：

```text
人物姓名 / JSON / CSV
          |
          v
资料检索或本地导入
          |
          v
事件抽取、去重、时间归一、来源与置信度标记
          |
          v
高德地点搜索与地理编码
          |
          v
统一 life-trace.json
          |
          +--> 高德 CLI / GUI 验证地点和地图操作
          |
          +--> HTML 生成器 --> 动态人生足迹地图
```

高德 CLI 用于 Agent 侧地图操作、地点核验和演示；高德 JS API 2.0 用于最终 HTML 的真实交互式底图。`AMAP_KEY` 必须是 Web 端（JS API）Key；新版 Key 还需对应的 `AMAP_SECURITY_KEY`。凭据缺失或鉴权失败时，页面显示明确错误，不得以 SVG 中国轮廓或案例底图冒充高德在线地图。

## 5. 组件边界

### 5.1 Skill 指令层

`SKILL.md` 负责判断输入模式并编排流程：

- “某人物的一生足迹”触发公开资料模式。
- “使用这个 JSON/CSV”触发上传模式。
- 先建立事件表，再补坐标，最后生成 HTML。
- 历史路线默认绘制地点关系弧线，不调用现代导航路径冒充史实。
- 输出必须包含资料来源和地点置信度。

### 5.2 数据规范层

统一数据文件包含：

- 人物元数据：名称、生卒年、简介、主题。
- 事件：唯一 ID、开始/结束时间、地点名、现代行政区、经纬度、事件标题、事件说明、阶段、来源列表、时间置信度、地点置信度。
- 路径：起点事件、终点事件、路径类型、是否争议。
- 生成信息：生成时间、坐标系、数据版本。

同一人物可在同一地点发生多次事件；地图可合并视觉节点，但时间轴不得丢失事件。

### 5.3 资料整理层

公开人物资料采用“两阶段”流程：

1. 先从百科、博物馆、政府文化页面、学术或权威历史资料中建立候选事件。
2. 对关键迁徙节点交叉核对；无法核对的内容保留为争议或排除。

每条事件至少保留一个来源 URL。重要转折点优先使用两个相互独立的来源。不得把搜索摘要当作最终来源。

置信度分为：

- `confirmed`：资料明确到时间和地点。
- `inferred`：资料明确到城市或区域，坐标采用代表性现代地标。
- `disputed`：来源冲突或地点存在多种说法。

### 5.4 高德适配层

- 安装官方 `amap-cli-skill` 与 `@amap-lbs/amap-gui`。
- 通过环境变量读取凭据，不把 Key 写入 Git 文件。
- 使用高德地点搜索或地理编码把现代地名转换为 GCJ-02 坐标。
- 缓存已确认坐标，减少重复请求并让 demo 可复现。
- 为古地名保留“历史名称”和“现代查询名称”两个字段。
- 使用 `AMap.Map` 渲染真实高德深色底图，使用 `AMap.Marker`、`AMap.Polyline` 和 `AMap.InfoWindow` 渲染节点、关系线与事件信息。
- `serve` 命令从本机环境变量读取 Key，在响应页面时注入运行配置；Key 不写入 `demo.html`、数据文件或 Git。

用户提供的 Key 已通过高德 CLI 的 `mapReady` 与 `mapState` 实测，可加载真实地图；它不出现在仓库、日志摘要或公开 demo 中。若后续启用新版安全校验，再配置独立的 `AMAP_SECURITY_KEY`。

### 5.5 HTML 生成与模板层

生成器读取 `life-trace.json`，把数据注入版本化模板。模板资源与人物数据分离，以便复用于不同人物。

页面布局：

- 左侧约 70%：真实高德深色地图、可缩放拖动的地图瓦片、红金迁徙线、发光节点和阶段标签。
- 右侧约 30%：人物标题、摘要、播放控制、年份、地点事件卡、来源和置信度。
- 窄屏：地图在上、时间轴在下，保留全部交互。

动态效果：

- 路线按时间逐段绘制，当前节点脉冲发光。
- 播放、暂停、重播和倍速控制。
- 点击时间轴聚焦地图节点；点击节点滚动并激活对应事件卡。
- 按阶段筛选后重新适配地图视野。
- `prefers-reduced-motion` 下关闭持续动画并保留状态变化。

页面启动流程：

1. `node life-trace.mjs serve demo.html` 启动仅绑定本机的预览服务。
2. 服务从 `AMAP_KEY` 和可选的 `AMAP_SECURITY_KEY` 读取凭据，并在内存中注入页面。
3. 浏览器加载高德 JS API 2.0，创建 `AMap.Map` 后再添加人物节点与路线。
4. 页面出现高德瓦片且 `AMap.Map` 实例可操作，才显示“高德地图已连接”；否则显示原始鉴权错误。

路线采用几何弧线或折线表达“先后关系”。只有当数据明确标记为现代实际出行且用户要求时，才使用高德路线规划。

## 6. 刘备 demo

demo 以可核验资料为准建立时间线，候选阶段包括涿郡早年、镇压黄巾与早期辗转、徐州、依附曹操与袁绍、荆州、入蜀、汉中、夷陵与白帝城。最终节点数量由资料核验决定，不为追求视觉密度制造事件。

demo 展示：

- 多阶段颜色或标签。
- 至少 10 个可点击事件。
- 至少 8 个不同地点节点。
- 至少一个推定地点，用于展示置信度表达。
- 至少一条来源冲突说明；若核验后不存在合适争议点，则不强行添加。
- 自动播放完整人生轨迹。

## 7. 输入与输出

### 输入

- 人物模式：人物姓名，可选时间范围、主题和详细程度。
- 上传模式：UTF-8 JSON 或 CSV。必需字段为时间、地点和事件说明；坐标可选。
- 凭据：运行环境中的 `AMAP_KEY` 和 `AMAP_SECURITY_KEY`。

### 输出

- 规范化人物数据 JSON。
- 地理编码缓存。
- 不含明文凭据的人物 HTML；通过 `serve` 命令加载高德运行库和本机环境变量。
- 数据质量报告，列出缺失坐标、推定地点、冲突和被排除条目。

## 8. 失败与降级

- 地名无法解析：保留事件，标记为未定位并从路线中暂时排除，质量报告给出修正建议。
- 多个同名地点：结合时代、行政区和相邻事件判断；仍不确定时要求确认或标记争议。
- 网络或高德接口失败：使用已有缓存生成，未缓存地点明确报错，不静默伪造坐标。
- 缺少或错误的 Web JS 凭据：页面显示配置提示与高德错误，不绘制案例底图，也不把未验证页面称为高德在线 demo。
- 资料时间不完整：支持年份、年月、日期和范围，排序时保留原始文本。
- 上传数据不合法：返回到行号/事件 ID 的错误，不生成半成品覆盖现有输出。

## 9. 隐私与发布安全

- 普通人模式只处理用户主动提供的数据。
- 默认不搜索普通人的私人资料。
- 发布前检查精确住址、手机号、证件号、未成年人信息和照片授权。
- `.gitignore` 排除 `.env`、本地凭据、临时输出和私人数据。
- 示例只包含公开历史资料。

## 10. 测试策略

### Skill 行为测试

按 `writing-skills` 的 RED-GREEN-REFACTOR 流程：先让独立 Agent 在没有本 Skill 指令的情况下处理代表性请求，记录其遗漏；再加入 Skill，验证其能选择正确模式、保留来源、避免伪造路线并生成规定输出。

代表性请求：

1. “生成刘备一生足迹图，并尽可能详细。”
2. “用我上传的 CSV 制作父亲的人生地图，不要联网搜索。”
3. “把李白从出生到去世的地点按现代驾车导航路线画出来。”

### 自动测试

- JSON/CSV 解析和字段校验。
- 时间归一、排序和同地点多事件处理。
- 置信度与来源校验。
- 地理编码结果缓存和失败行为。
- HTML 转义、数据注入和必要 DOM 元素。
- `serve` 只从环境变量注入高德配置，生成文件不包含 Key。
- 生成 HTML 必须包含真实高德容器与 `AMap.Map`、`AMap.Marker`、`AMap.Polyline` 初始化逻辑，不得包含 SVG 中国轮廓降级图。
- 不含明文 Key 的仓库扫描。

### 集成与视觉验证

- 官方 CLI 能显示帮助、启动并报告状态。
- 使用有效凭据完成至少一次高德 CLI `mapState` 操作，并核对返回的中心、缩放和样式。
- 用 `serve` 打开刘备 demo，确认高德瓦片已加载且页面存在可操作的 `AMap.Map` 实例。
- 逐项检查真实地图上的路线播放、标记/时间轴联动、信息窗、来源卡片和窄屏布局。
- 对 demo 截图留作发布展示与视觉回归证据。

## 11. 目录设计

目标项目目录名为 `life-trace`：

```text
life-trace/
├── SKILL.md
├── agents/openai.yaml
├── scripts/
│   ├── build-life-trace.mjs
│   ├── validate-life-trace.mjs
│   └── geocode-amap.mjs
├── assets/
│   └── template/
├── references/
│   ├── data-schema.md
│   └── amap-integration.md
├── examples/
│   ├── liu-bei.json
│   └── personal-template.csv
├── tests/
└── demo/
    └── liu-bei.html
```

`SKILL.md` 保持精简，详细数据规范和高德命令放入按需读取的 references；可执行逻辑放入 scripts；模板放入 assets。GitHub 项目说明和发布截图是否加入根目录，以 ClawHub/GitHub 实际发布要求为准，避免在 Skill 内堆放无关文档。

## 12. 验收标准

以下条件必须全部满足：

1. 当前文件夹中存在独立 `life-trace` 目录。
2. 官方高德 Skill 和 CLI 已安装，版本/帮助命令可验证。
3. 用户 Key 通过环境变量配置且不进入仓库；缺少安全密钥时有明确状态。
4. Skill 包含有效 `SKILL.md` 与 `agents/openai.yaml`，通过官方快速校验。
5. 名人模式和普通人上传模式均有明确、可执行流程。
6. 刘备数据包含来源与置信度，并通过数据校验。
7. HTML demo 通过 `serve` 打开后加载真实高德地图瓦片，显示动态路线、可点击标记和同步侧栏；不得用 SVG 案例地图代替。
8. 自动测试全部通过，浏览器视觉检查完成。
9. 文档明确给出 Skill 名称、使用场景、功能描述、使用教程和 demo 演示步骤。
10. 仓库具备 GitHub 与 ClawHub 发布所需文件，且不存在明文凭据或私人数据。

## 13. 实施原则

- 先测试再实现；每项功能都先观察预期失败。
- 只实现本规格列出的能力，不增加账号、数据库或在线编辑器。
- 史料准确性优先于路线密度。
- 高德在线能力和视觉模板分别验证，任何降级都必须显式说明。
